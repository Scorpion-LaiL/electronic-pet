## 1. 项目概述

### 1.1 项目简介

本项目旨在从 0 到 1 开发一款受拓麻歌子启发的轻量级电子宠物养成游戏。玩家通过日常照料、互动和观察成长，与虚拟宠物建立持续性的情感连接。项目首期采用网页端实现，以便快速验证玩法、加快迭代速度，并降低技术与发布门槛。

### 1.2 项目愿景

打造一款具备轻陪伴、低负担、易上手特征的宠物养成游戏，让玩家在碎片化时间内也能体验到照顾、反馈和成长的乐趣，并唤起 90 后用户对于童年电子宠物时代的情感记忆。

### 1.3 一句话定义

做一款可在网页中游玩的轻量电子宠物养成游戏，玩家每天花几分钟照顾宠物，并通过不同照顾方式触发状态变化与成长反馈。

## 2. 项目背景与立项目的

### 2.1 项目背景

拓麻歌子类产品之所以长期有吸引力，在于其玩法结构简单但反馈闭环清晰，天然适合碎片化体验和长期陪伴。对于独立开发或 AI coding 驱动的项目来说，这类游戏也具备以下优势：

- 核心循环明确，便于快速构建最小可玩版本。
- 规则以状态变化和交互反馈为主，适合模块化开发和测试。
- 前期可先用占位美术完成玩法验证，不依赖完整美术资源即可推进。
- 后续扩展空间大，可逐步加入多形态成长、事件系统、情绪互动、装扮或 AI 对话能力。

除玩法价值之外，拓麻歌子对于 90 后人群还承载着明显的时代记忆。它不仅是一个“要照顾的小宠物”，也是很多人童年关于掌机、像素、陪伴和日常惦记的情感符号。因此，本项目不仅要复现电子宠物的核心循环，也要在体验气质上保留怀旧、亲切和熟悉感。

### 2.2 立项目的

本项目的立项目的包括：

- 以标准开发流程完成一个从需求到交付的完整游戏项目样例。
- 通过 AI coding 的方式验证“文档驱动 + 模块化实现 + 测试回归”的协作方法。
- 在可控范围内做出一款具备独立可玩性的宠物养成游戏 MVP。
- 为后续加入更复杂的玩法系统、美术包装或 AI 陪伴能力打下结构基础。

## 3. 项目目标

### 3.1 业务目标

- 完成一个可运行、可试玩、可持续迭代的宠物养成游戏项目。
- 建立一套适用于 AI coding 的标准化项目文档体系。
- 形成从立项、需求、设计、开发、测试到发布复盘的完整交付链路。

### 3.2 产品目标

- 提供一个易理解、低门槛的养成体验。
- 让玩家能够通过喂食、玩耍、清洁、休息等操作影响宠物状态。
- 让宠物能够随时间变化并给出可感知反馈。
- 实现基础成长机制，使玩家感受到照顾行为带来的长期结果。
- 在整体体验、视觉语言和互动反馈上唤起 90 后用户对童年电子宠物的情感联想。

### 3.3 首版目标（MVP）

首版聚焦最小可玩闭环，目标是在不引入后端和复杂联网能力的前提下完成以下能力：

- 进入游戏后可以创建或接收一只初始宠物。
- 玩家可以为宠物起名字并选择性别。
- 玩家可以执行基础照顾操作。
- 宠物状态会随时间变化。
- 宠物具备至少 3 个成长阶段。
- 游戏支持本地存档与恢复。
- 游戏具备基础状态反馈与提示机制。

## 4. 目标用户

### 4.1 核心用户画像

首版面向以下类型玩家：

- 喜欢轻量养成和治愈陪伴体验的休闲玩家。
- 偏好低压力、短时互动节奏的用户。
- 对电子宠物、像素风或怀旧型轻游戏有兴趣的玩家。
- 对拓麻歌子、掌上电子宠物或童年养成玩具有记忆和情感连接的 90 后用户。
- 愿意每天投入 1 到 5 分钟完成简单互动的用户。

### 4.2 用户需求判断

目标用户更看重的不是复杂策略深度，而是以下体验：

- 宠物“像在活着”一样持续变化。
- 自己的照顾行为会产生明确反馈。
- 游戏不会要求高频打卡，也不会带来过强惩罚。
- 每次打开游戏都能看到一点新变化或新的照顾目标。
- 在操作、氛围和反馈中重新感受到童年电子宠物带来的熟悉感。

## 5. 产品定位

### 5.1 产品类型

单机网页宠物养成游戏。

### 5.2 核心价值

- 陪伴感：宠物持续存在，玩家能建立情感连接。
- 成长感：照顾方式影响宠物状态和成长结果。
- 轻反馈：每次短暂操作都能看到清晰变化。
- 低压力：首版采用相对温和的失败机制，避免强挫败感。
- 怀旧感：通过经典电子宠物式的互动结构和氛围设计，唤起 90 后用户的童年记忆。

### 5.3 差异化方向

在保持拓麻歌子式经典结构的同时，项目可优先探索以下差异化方向，为后续版本保留空间：

- 更现代的网页交互与视觉反馈。
- 在怀旧感和现代易用性之间取得平衡，不做纯复刻，而是做“记忆重现 + 体验升级”。
- 更友好的低惩罚养成节奏。
- 更清晰的状态可视化与事件提示。
- 后续扩展为带 AI 对话或情绪陪伴能力的宠物系统。

## 6. 项目范围

### 6.1 首版纳入范围（In Scope）

MVP 计划纳入以下内容：

- 单只宠物养成体验。
- 基础状态系统。
- 基础交互系统。
- 时间推进与状态衰减机制。
- 基础成长阶段机制。
- 本地存档与恢复。
- 基础 UI 界面与反馈提示。
- 核心逻辑测试与手工体验测试。

### 6.2 首版建议包含的核心功能

- 宠物展示区域。
- 宠物身份设定（名字、性别）。
- 状态栏展示。
- `喂食` 功能。
- `玩耍` 功能。
- `清洁` 功能。
- `睡觉/休息` 功能。
- 状态随时间变化。
- 成长阶段变化提示。
- 异常状态提示，例如疲劳或生病。
- 本地自动存档。

### 6.3 首版暂不纳入范围（Out of Scope）

为保证节奏和交付质量，以下内容不进入首版：

- 联网系统。
- 多宠物并行养成。
- 玩家社交或拜访系统。
- 复杂商城、货币和任务系统。
- 大量装扮内容。
- 后端账户体系。
- 多语言版本。
- 复杂剧情线。
- 高复杂度美术生产流程。

## 7. 成功标准

### 7.1 项目成功标准

如果首版达到以下条件，则视为项目第一阶段成功：

- 已形成完整的项目文档链路，至少包括章程、需求分析、PRD、技术设计、测试计划。
- 已完成可运行的网页游戏版本。
- 玩家可以独立完成核心养成循环。
- 关闭页面后重新进入，宠物状态能够恢复。
- 主流程中不存在阻塞游戏继续进行的严重缺陷。

### 7.2 体验成功标准

- 首次进入的玩家能在 1 分钟内理解基本玩法。
- 玩家能在 3 分钟内完成至少一次完整互动闭环。
- 宠物的状态变化对玩家来说是可见且可理解的。
- 游戏在短时游玩中能提供基本的陪伴和成长反馈。

### 7.3 工程成功标准

- 核心逻辑与 UI 展示分层清晰。
- 关键状态更新逻辑具备可测试性。
- 代码结构支持后续继续增加系统功能。
- 至少覆盖核心状态计算和关键交互的基础测试。

## 8. 项目约束与假设

### 8.1 约束条件

- 项目采用 AI coding 协作为主要开发方式之一。
- 首版优先选择前端单机架构，避免后端依赖。
- 开发资源有限，首期以玩法跑通优先，不追求完整商业化包装。
- 视觉资源可先采用占位风格或简单自制资源。

### 8.2 基本假设

- 首版用户愿意接受轻量内容和基础画面表现。
- 经典电子宠物玩法在现代网页环境下依然具备可验证价值。
- 通过合理拆分文档和任务，可以让 AI coding 在项目中发挥较高效率。

## 9. 主要风险与应对策略

| 风险 | 说明 | 应对策略 |
| --- | --- | --- |
| 范围膨胀 | 容易过早加入太多系统 | 严格以 MVP 为边界，新增功能进入后续版本池 |
| 玩法单薄 | 互动过少会降低留存 | 在首版中确保状态反馈、成长反馈和事件提示足够清晰 |
| 数值失衡 | 状态下降过快或过慢都会影响体验 | 在需求和测试阶段单独编写数值规则与试玩反馈 |
| AI 生成结构混乱 | 代码可能出现职责不清 | 在开发前先完成技术设计与模块边界定义 |
| 存档逻辑缺陷 | 时间推进和恢复可能出错 | 设计独立的存档模型，并优先测试时间结算逻辑 |
| 反馈不足 | 玩家可能不知道行为是否生效 | 强化按钮反馈、消息提示和状态变化展示 |

## 12. 角色分工建议

在当前阶段，项目可按以下角色思路推进：

- 产品角色：负责目标定义、需求取舍、优先级判断。
- 技术角色：负责架构、代码实现、测试与质量控制。
- AI 协作角色：负责文档初稿、任务拆解、编码支持、测试建议与迭代辅助。

如果当前项目由个人主导，则可理解为一个人同时承担产品与技术决策，AI 作为协作开发助手。

## 13. 下一阶段说明

项目章程确认后，下一步进入 `需求分析` 阶段，重点完成以下内容：

- 明确目标用户与使用场景。
- 定义核心玩法循环。
- 识别功能需求与非功能需求。
- 给出 MVP 功能边界和优先级。
- 为后续 PRD 编写提供输入。

## 14. 当前版本待确认项

以下内容建议在下一轮文档中继续细化和确认：

- 游戏整体风格定位是否更偏怀旧像素、现代治愈，还是极简图形化。
- 首版是否引入“生病但不死亡”的低惩罚机制。
- 宠物成长阶段数量是否固定为 3 段。
- 首版是否加入随机事件系统。
- 首版目标平台是否仅限网页端。

## 15. 结论

本项目将以“轻量、可玩、可迭代”的电子宠物养成游戏为目标，采用标准化文档流程和 AI coding 协作方式推进。首版聚焦最小可玩闭环，通过明确范围、控制复杂度和重视测试，为后续扩展玩法与体验建立稳定基础。
