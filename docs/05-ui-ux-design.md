## 1. 文档目标

本文档用于定义首版产品的界面结构、信息层级、交互逻辑和体验风格，重点回答以下问题：

- 玩家进入游戏后首先看到什么
- 主界面应该如何组织信息和操作
- 状态、成长、危险、死亡等关键信息如何被感知
- 怀旧感如何通过界面、文案和反馈设计体现
- 如何在桌面和移动端都保持清晰、可玩和有氛围的体验

本文档聚焦 `交互结构和界面方向`，不在本阶段输出高保真视觉稿。

## 2. 设计原则

## 2.1 核心原则

首版 UI/UX 设计遵循以下原则：

- `宠物优先`：宠物本体必须是视觉中心
- `单屏闭环`：核心状态查看和操作尽量在同一屏完成
- `直观反馈`：每次操作都应被立即感知
- `轻怀旧`：保留电子宠物时代的熟悉感，但不过度复古到影响理解
- `低负担`：不堆砌复杂信息，不制造过强操作压力

## 2.2 怀旧与现代的平衡

界面设计不是直接复刻九十年代设备，而是提炼其情绪结构：

- 保留“小屏幕里有一个被你照顾的小生命”的感觉
- 保留按钮操作的简洁感和仪式感
- 保留带一点玩具气质的界面氛围

同时升级以下体验：

- 状态信息更容易理解
- 操作路径更短
- 提示更清晰
- 存档、危急状态、死亡流程更容易被解释

## 3. 信息架构

## 3.1 首版页面结构

首版建议采用 `少页面、主界面驱动` 的结构，核心页面如下：

1. `欢迎/进入页`
2. `宠物身份设定页/弹层`
3. `主养成页`
4. `成长提示弹层`
5. `危急提示层`
6. `死亡回顾页`
7. `历史记录/纪念入口`（可选，首版可先简化为局部模块）

## 3.2 页面优先级

首版真正常驻的核心是：

- 主养成页

其余页面更适合作为 `状态性页面` 或 `覆盖层`：

- 欢迎页：首次进入时出现
- 身份设定页：首次进入时用于命名和选择性别
- 成长提示：阶段变化时弹出
- 危急提示：进入危急状态时突出显示
- 死亡回顾页：死亡时单独承载情绪反馈

## 4. 用户流程设计

## 4.1 首次进入流程

建议流程：

1. 用户进入游戏
2. 展示欢迎信息或简短引导
3. 进入宠物身份设定流程
4. 用户输入宠物名字并选择性别
5. 呈现初始宠物
6. 用最少提示告诉用户可以进行 `喂食 / 玩耍 / 清洁 / 休息`
7. 引导用户完成第一次互动

设计要求：

- 首次进入应尽量轻，不做复杂教程
- 命名和性别选择应简洁完成，不让用户被表单感打断
- 引导应像“带用户认识宠物”，而不是像系统培训

## 4.1A 宠物身份设定流程

建议流程：

1. 展示宠物初始形象或轮廓
2. 提供名字输入框
3. 提供性别选择控件
4. 用户确认后进入主界面

设计要求：

- 名字输入框应清楚但不过长
- 性别选择应简单直接
- 整个流程应像“认识它”，而不是填写资料卡

## 4.2 日常进入流程

建议流程：

1. 用户进入游戏
2. 直接看到主养成页
3. 若有离线结算结果，则以轻提示方式展示
4. 用户查看状态并做出操作
5. 自动保存

设计要求：

- 老用户不应被多余页面打断
- 回来就能立刻看到宠物，很重要

## 4.3 危急状态流程

建议流程：

1. 系统检测进入危险或危急状态
2. 主界面加强视觉提示
3. 显示短文案告诉用户当前问题
4. 鼓励用户立即采取补救操作
5. 若恢复成功，提示宠物状态稳定

设计要求：

- 要有足够压迫感，但不能乱
- 必须让用户知道“该做什么”

## 4.4 死亡与重新开始流程

建议流程：

1. 宠物进入死亡
2. 主界面过渡到死亡回顾页
3. 展示宠物本轮历程摘要
4. 给用户一个简短告别时刻
5. 提供 `重新开始` 入口
6. 开启下一轮养成

设计要求：

- 要有情绪落点
- 不能机械跳回首页
- 重新开始按钮要明确，但不要破坏回顾时刻

## 5. 主养成页设计

## 5.1 主界面目标

主养成页是整个产品的核心，需要同时承担以下职责：

- 展示宠物当前状态
- 承载全部核心操作
- 传递宠物情绪和生命感
- 告诉用户当下最需要处理的事

## 5.2 主界面信息层级

建议按以下优先级组织信息：

1. `宠物本体`
2. `关键状态`
3. `核心操作`
4. `提示信息`
5. `成长/时间信息`

这意味着用户第一眼看到的不是一堆数字，而是宠物本身。

## 5.3 主界面布局建议

建议采用纵向单页布局，可适配桌面与移动端。

推荐结构：

### 区域 A：顶部信息区

内容建议：

- 游戏标题或宠物世界名称
- 宠物名字
- 宠物性别
- 当前时间段或轻环境信息
- 设置/重置等次级入口

设计原则：

- 保持轻量，不要压过宠物主体

### 区域 B：宠物展示区

内容建议：

- 宠物主形象
- 当前动作或表情
- 简短状态泡泡

设计原则：

- 这是整个页面的视觉中心
- 宠物应占据足够空间，不能被 UI 挤得像图标
- 成长变化、危急状态、睡眠状态都应优先通过这里感知

### 区域 C：状态面板区

内容建议：

- 饥饿
- 心情
- 清洁
- 体力
- 健康

设计原则：

- 用进度条、图标和颜色共同表达
- 不推荐只显示数字
- 状态项数量要让用户一眼能扫完

### 区域 D：操作区

内容建议：

- 喂食按钮
- 玩耍按钮
- 清洁按钮
- 休息按钮

设计原则：

- 四个核心按钮要平级清楚
- 点击区域足够大
- 可结合像玩具按键的设计语言强化怀旧感

### 区域 E：提示与日志区

内容建议：

- 当前状态提示
- 离线结算提示
- 成长提示
- 恢复或异常提示

设计原则：

- 信息要短
- 可滚动历史不应喧宾夺主
- 重要提示优先即时展示，次要提示可收纳

### 区域 F：成长与纪念信息区

内容建议：

- 当前成长阶段
- 存活时长
- 照顾评价摘要

设计原则：

- 作为长期目标信息存在
- 不要比当前状态更抢眼

## 6. 状态展示设计

## 6.1 展示方式建议

每个状态建议采用以下组合表达：

- 图标
- 名称
- 进度条
- 状态文案

例如：

- 饥饿：小餐具图标 + 进度条 + “有点饿了”
- 心情：笑脸图标 + 进度条 + “今天心情不错”

## 6.2 颜色层级建议

建议统一采用 4 档状态颜色：

- 良好：柔和绿色
- 正常：暖黄色或米色
- 危险：橙色
- 危急：红色

设计要求：

- 颜色必须辅助信息理解，但不能只依赖颜色
- 文字或图标变化要同时存在，兼顾可访问性

## 6.3 状态文案策略

避免只写“数值下降”。建议使用更有生命感的表达：

- “它有点饿了”
- “今天看起来挺开心”
- “身上有点脏脏的”
- “它有些累了”
- “看起来不太舒服”

设计意图：

- 让用户感觉是在照顾一个角色，而不是维护一个数据面板

## 7. 核心操作交互设计

## 7.1 按钮设计原则

四个核心操作按钮需要满足：

- 一眼能认出用途
- 点击后立即有反馈
- 在桌面和移动端都易点
- 视觉上带一点掌机/玩具按钮的仪式感

## 7.2 操作反馈层次

每次点击操作后建议至少出现以下反馈中的 2 种：

- 按钮按压反馈
- 宠物动作或表情变化
- 状态条变化
- 一句短文案提示

例如喂食后可出现：

- 宠物咀嚼或开心动作
- 饥饿条上升
- 文案：“它吃得很香”

## 7.3 不可用状态提示

当某按钮暂时不可用时，不应只做灰掉处理，还应提供原因说明。

例如：

- “它正在睡觉”
- “它太累了，不想玩”
- “现在已经很干净啦”

## 8. 成长、危险与死亡的交互表现

## 8.1 成长提示设计

成长是高情绪时刻，不应只在角落里变一行字。

建议表现：

- 宠物展示区出现阶段变化动画
- 中央弹出成长提示层
- 文案带一点纪念感

建议内容：

- 新阶段名称
- 简短成长说明
- 一句情绪化文案

## 8.2 危险状态表现

危险状态应在主界面上即可明显感知。

建议表现：

- 状态面板出现高亮警示
- 宠物动作明显变弱
- 提示区文案升级
- 页面局部氛围变得更紧张，但不应导致阅读困难

## 8.3 危急状态表现

危急状态需要比危险状态再强一层。

建议表现：

- 危急提示条或弹层出现
- 宠物展示区出现更强烈的虚弱状态
- 主操作区附近出现补救建议

设计重点：

- 不只是告诉用户“很危险”
- 更要告诉用户“现在做什么最有效”

## 8.4 死亡回顾页设计

死亡回顾页是首版最重要的情绪页面之一。

目标：

- 让用户感受到一段旅程结束了
- 让死亡不只是失败结算
- 保留怀旧感中的一点遗憾和纪念意味

建议页面结构：

1. 宠物最终形象或纪念图
2. 告别文案
3. 宠物名字与性别
4. 存活天数
5. 达到的成长阶段
6. 简单照顾评价
7. `重新开始` 按钮

文案方向建议：

- 温柔
- 克制
- 不夸张
- 有一点童年回忆感

避免：

- 过于沉重
- 过于冰冷
- 像系统报错页

## 9. 怀旧风格方向

## 9.1 整体风格关键词

建议首版 UI 风格围绕以下关键词：

- `掌机感`
- `像素感`
- `玩具感`
- `温柔怀旧`
- `轻治愈`

## 9.2 视觉语言建议

可参考的方向：

- 类小屏幕界面边框
- 低分辨率灵感但现代清晰呈现
- 柔和米白、浅绿、暖黄、雾粉等复古友好色
- 圆角、塑料按键感、轻阴影

不建议的方向：

- 纯赛博科技风
- 纯极简企业后台风
- 过深色、过锐利、过冷的氛围

## 9.3 文案风格建议

文案应符合以下气质：

- 简短
- 温柔
- 有点像在和宠物说话
- 带一点点童年语感，但不要幼稚过头

示例方向：

- “它今天精神不错”
- “好像有点想你了”
- “它已经长大一点点了”

## 11. 响应式与可用性要求

## 11.1 移动端要求

- 单手可触达核心操作区
- 状态条和按钮足够大
- 重要提示不被折叠到底部看不见

## 11.2 桌面端要求

- 保持页面中心聚焦，不要铺得太散
- 宠物展示区仍应占主视觉
- 空间变大时，优先强化氛围而不是塞更多信息

## 11.3 可访问性要求

- 颜色不是唯一状态表达方式
- 文案需要简洁直白
- 按钮状态变化要足够清楚
- 危急和死亡提示要有明显层级差异

## 12. 动效与声音建议

## 12.1 动效原则

- 动效应服务于生命感和反馈感
- 不追求花哨，追求“它真的在回应你”

建议优先级：

1. 操作反馈动效
2. 状态变化动效
3. 成长动效
4. 危急状态提示动效

## 12.2 声音建议

若后续加入音效，建议风格偏：

- 电子玩具感
- 简短提示音
- 低复杂度、可记忆的小声音

避免：

- 过于写实
- 过于吵闹
- 与怀旧氛围不符的现代 UI 音

## 13. 首版 UI 范围控制

为保证开发效率，首版 UI 建议聚焦于：

- 一个高完成度主界面
- 一套明确状态展示
- 四个核心操作按钮
- 一套成长提示
- 一套危急提示
- 一个死亡回顾页

以下内容可后置：

- 复杂图鉴页面
- 丰富主题皮肤
- 多层设置系统
- 大量过场动画

## 15. 结论

首版交互与 UI 设计方向已经明确：以 `宠物为视觉中心、单屏完成核心闭环、用状态和反馈塑造生命感、用怀旧气质连接 90 后记忆` 为核心原则，构建一个既像电子宠物、又比传统电子宠物更清晰友好的现代网页体验。下一步可进入技术设计文档，将这些界面与规则转化为实际工程结构。
