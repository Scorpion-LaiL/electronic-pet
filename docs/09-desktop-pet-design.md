# 桌面宠物化设计文档（Desktop Pet Design）

## 1. 目标

把当前的 `AI Coding Pet` 从“打开网页后照顾的电子宠物”升级成“常驻电脑桌面、会移动、会打扰你一下、但不至于过度侵入”的桌面宠物。

这次改造的重点不是重写养成规则，而是把现有的：

- 宠物状态系统
- 时间流逝与离线结算
- 喂食 / 玩耍 / 清洁 / 休息
- 死亡与重新开始

迁移到新的 `桌面存在形态` 中。

## 2. 体验方向

新的桌面宠物版本建议遵循这几个体验原则：

- `宠物常驻`：用户不需要先进入一个完整主界面，宠物默认就在桌面边缘活动。
- `轻打扰`：状态不好时会提示，但不能频繁打断工作。
- `主角是宠物，不是面板`：面板默认收纳，宠物本体始终是视觉中心。
- `行为有性格`：高心情时更活跃，低体力时移动变慢，睡觉时会缩到角落。
- `随手照顾`：点击宠物或托盘菜单就能快速喂食、玩耍、清洁、休息。

## 3. 产品形态设计

## 3.1 主体形态

建议拆成两个层级：

1. `桌面宠物层`
2. `管理面板层`

### 桌面宠物层

- 透明背景
- 无边框
- 始终置顶
- 可拖拽
- 默认只显示宠物本体、气泡、少量特效

### 管理面板层

- 点击宠物、右键宠物或托盘图标后展开
- 展示状态条、日志、成长记录
- 支持收起

这和当前仓库的新原型方向一致：

- 宠物主视图单独放大
- 状态和动作被收纳到侧边照顾面板

## 3.2 宠物在桌面上的行为

建议首版桌面化只做 `有限行为`，不要一开始就追求复杂 AI。

### 空闲行为

- 沿桌面底部小范围巡游
- 在左右边缘停留
- 偶尔回头、跳一下、停顿

### 状态联动

- `心情高`：移动速度更快，随机跳跃更多
- `体力低`：移动频率下降，步幅缩短
- `清洁低`：出现不高兴或脏兮兮提示
- `危险/危急`：气泡和轮廓发出警示
- `睡眠中`：进入固定角落，切换睡眠动作

### 用户交互

- 单击：展开或收起照顾面板
- 拖拽：改变宠物停靠位置
- 右键：打开快捷菜单
- 双击：打开完整管理面板

## 4. 技术方案建议

## 4.1 推荐路线

基于当前仓库的技术背景，推荐第二阶段采用：

- `React + TypeScript` 继续保留前端 UI 和状态逻辑
- `Electron` 作为桌面容器

推荐 Electron 的原因：

- 这套项目已经是纯前端工程，接入成本低
- 透明窗口、无边框窗口、置顶窗口能力成熟
- IPC、托盘、开机启动、窗口控制都比较直接
- 不需要先补 Rust 工具链

## 4.2 桌面容器结构

建议新增如下结构：

```text
electron/
  main.ts
  preload.ts
src/
  desktop/
    motion/
    window/
    tray/
```

职责划分：

- `React UI`
  负责宠物显示、状态条、面板、动画反馈
- `Electron Main`
  负责窗口创建、托盘、开机启动、始终置顶、位置控制
- `Preload + IPC`
  负责把“窗口移动 / 点击穿透 / 桌面边缘吸附”等能力暴露给前端

## 4.3 窗口设计

建议至少拆成两个窗口：

### 1. `pet-window`

- 透明
- 无边框
- 始终置顶
- 不显示任务栏图标
- 默认尺寸较小
- 只承载宠物本体与轻提示

### 2. `panel-window`

- 普通半透明面板
- 由宠物窗口触发显示
- 可固定在宠物旁边或单独浮出

首版也可以只用一个窗口完成，但长期维护上，双窗口更清晰。

## 5. 代码层改造建议

## 5.1 保持不动的部分

这些模块基本可以直接复用：

- `src/domain/**`
- `src/services/game-engine.ts`
- `src/services/local-storage.ts`
- `src/store/game-store.ts`

也就是说，`养成规则` 不需要为了桌面化而重写。

## 5.2 需要新增的状态

当前的宠物状态还缺少一些桌面行为字段，建议新增一个独立的 `desktop runtime`，不要直接污染养成状态。

建议新增：

```ts
type DesktopPetState = {
  position: { x: number; y: number };
  velocity: { x: number; y: number };
  facing: 'left' | 'right';
  anchor: 'bottom-left' | 'bottom-right' | 'free';
  movementMode: 'idle' | 'wander' | 'sleep' | 'alert';
  panelOpen: boolean;
  clickThrough: boolean;
  lastMovedAt: number;
};
```

这样可以保证：

- 养成状态负责“它过得好不好”
- 桌面状态负责“它现在怎么在桌面上表现”

## 5.3 新模块建议

建议新增：

- `src/desktop/motion/pet-motion-engine.ts`
- `src/desktop/window/window-bridge.ts`
- `src/desktop/tray/tray-actions.ts`
- `src/hooks/use-desktop-pet.ts`

职责分别是：

- 运动规则
- 与 Electron 窗口通信
- 托盘和快捷动作
- 把桌面表现接到 React

## 6. 分阶段落地方案

## 阶段 A：桌面宠物 UI 原型

目标：

- 先把网页主界面改造成“宠物主视图 + 收纳面板”的结构
- 验证视觉方向和交互节奏

当前仓库已经开始往这个方向调整。

## 阶段 B：接入 Electron 外壳

目标：

- 跑起透明无边框窗口
- 把宠物挂到桌面上
- 保留现有状态与存档逻辑

交付点：

- `npm run desktop:dev`
- 宠物窗口可显示、拖拽、置顶

## 阶段 C：加入桌面行为

目标：

- 在桌面底部巡游
- 状态驱动移动模式
- 展开和收起照顾面板

交付点：

- 宠物会走动
- 睡觉与危急状态有明显差异
- 快捷操作可用

## 阶段 D：系统级增强

可追加：

- 托盘菜单
- 开机启动
- 点击穿透
- 多显示器支持
- 通知提醒节流

## 7. 这次设计对应当前仓库的含义

当前仓库最值得保留的是“规则层已经干净分离”。

所以改造策略应该是：

1. `保留 domain 和 game engine`
2. `重做主视图结构`
3. `新增 desktop runtime`
4. `最后接入 Electron`

这样改造风险最低，也最适合你现在这个项目的阶段。
