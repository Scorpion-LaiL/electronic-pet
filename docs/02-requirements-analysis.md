## 1. 文档目标

本文档用于在项目章程的基础上，进一步明确本项目的用户需求、使用场景、核心玩法、功能需求、非功能需求和 MVP 范围。本文档的目标不是定义技术实现细节，而是回答以下问题：

- 这款游戏要解决什么体验需求。
- 目标用户在什么场景下会使用它。
- 玩家在游戏中会进行哪些核心行为。
- 首版必须具备哪些功能，哪些功能可以后置。
- 后续 PRD、玩法设计和技术设计应该围绕什么展开。

## 2. 需求背景

### 2.1 项目来源

本项目希望参考拓麻歌子类电子宠物游戏的核心结构，构建一个适合现代网页环境、适合 AI coding 协作开发的宠物养成游戏。与传统复杂游戏项目相比，本项目首版更关注：

- 玩法闭环是否成立。
- 状态反馈是否清晰。
- 陪伴感和成长感是否能够被用户感知。
- 文档驱动开发流程是否能够顺利支撑从 0 到 1 的实现。

除此之外，本项目还有一层重要目标，即为 90 后用户提供一种“重新摸到童年记忆”的体验。拓麻歌子及类似电子宠物并不只是一个玩法原型，它承载的是一代人在童年时期对于掌上玩具、像素式反馈、照顾责任感和日常惦记的共同记忆。因此，本项目在需求上不仅要满足“能玩”，还要满足“能唤起熟悉感”。

### 2.2 需求分析范围

本文档聚焦于首版 MVP 的需求分析，主要涵盖：

- 用户价值与体验目标
- 用户角色与场景
- 核心玩法循环
- 功能需求与规则边界
- 非功能需求
- MVP 功能优先级

本文档暂不覆盖：

- 详细 UI 样式规范
- 详细数值表
- 详细技术架构设计
- 发布渠道和商业化方案

## 3. 用户需求分析

### 3.1 目标用户画像

本项目首版面向以下核心用户群体：

1. 喜欢治愈、轻陪伴体验的休闲玩家
2. 喜欢电子宠物、养成类、怀旧类小游戏的玩家
3. 对拓麻歌子或童年电子宠物有情感记忆的 90 后人群
4. 希望在碎片化时间中获得短时互动反馈的用户
5. 对复杂规则、重度策略和高压力在线任务不感兴趣的用户

### 3.2 用户核心诉求

目标用户进入这类游戏时，通常并不是为了挑战高难度玩法，而是为了获得以下体验：

- 有一个“持续存在”的小宠物可以照顾
- 自己的行为能立刻影响宠物状态
- 打开游戏时能看到变化、反馈和成长
- 不需要投入很长时间也能感到有参与感
- 即使短暂离开，也不会因为惩罚过重而失去继续玩的动力
- 在视觉气质、交互节奏和反馈方式上感受到童年电子宠物的熟悉氛围
- 通过现代设备重新获得一种“小时候养过它”的回忆感

### 3.3 用户痛点与机会点

现有同类体验常见的问题包括：

- 玩法过于单一，互动反馈弱
- 惩罚过重，容易让用户因错过照顾时间而放弃
- 界面信息不够清楚，用户不知道该做什么
- 需要频繁上线，形成打卡压力

本项目的机会点在于：

- 用更清晰的界面和状态提示降低理解成本
- 用更友好的节奏设计降低放弃率
- 用更现代的网页交互提高反馈感
- 用适度怀旧的视觉、文案和交互语言连接 90 后用户的童年记忆
- 为后续 AI 对话或事件扩展预留空间

## 4. 用户场景与使用路径

### 4.1 典型使用场景

#### 场景 A：首次进入

用户第一次打开游戏，希望快速理解“这是什么、我能做什么、我现在应该点哪里”。此时需求重点是：

- 进入门槛低
- 界面主要元素清楚
- 第一次互动容易成功

#### 场景 B：日常短时打开

用户在一天中的碎片时间再次打开游戏，希望快速查看宠物状态，并完成一次或几次照顾行为。此时需求重点是：

- 状态变化一眼可见
- 核心按钮操作直接
- 一次互动能获得清晰反馈

#### 场景 C：较长时间未打开后回归

用户隔了一段时间再次进入，希望了解离线期间发生了什么，并决定如何补救或继续照顾。此时需求重点是：

- 离线结算结果清楚
- 不因一次离开就完全失去进度
- 有合理的恢复空间

#### 场景 D：观察成长变化

用户在连续照顾一段时间后，希望看到宠物成长、变化或新的表现形式。此时需求重点是：

- 成长条件清晰
- 成长结果可感知
- 玩家能理解“为什么会长成这样”

### 4.2 用户基本使用路径

首版的核心使用路径可以概括为：

1. 进入游戏
2. 查看宠物状态
3. 识别当前最需要处理的问题
4. 执行基础互动操作
5. 看到状态与表现变化
6. 游戏自动存档
7. 后续再次进入并继续照顾

## 5. 核心玩法循环分析

### 5.1 核心玩法定义

本项目的核心玩法循环是一个典型的“状态衰减 -> 玩家干预 -> 状态恢复 -> 成长反馈 -> 持续照料”的闭环。

### 5.2 核心循环步骤

可以将核心循环拆解为：

1. 宠物随时间发生状态变化
2. 玩家进入游戏并查看状态
3. 玩家根据状态执行喂食、玩耍、清洁或休息等操作
4. 操作结果即时反馈到数值和表现上
5. 宠物在累计照顾下产生成长变化
6. 成长变化激励玩家继续下一轮照顾

### 5.3 核心循环成立条件

如果要让这个循环真正可玩，至少需要满足以下条件：

- 状态变化必须足够明显，玩家能理解问题所在
- 玩家操作必须足够直接，不需要复杂学习成本
- 每个操作都必须有可感知结果
- 成长反馈不能太慢，否则长期动力不足
- 惩罚不能过重，否则玩家容易中途放弃

## 6. 需求边界定义

### 6.1 本项目要满足的核心体验需求

首版必须优先满足以下体验：

- 用户能在短时间内理解并完成一次照顾循环
- 宠物状态变化具有“活着”的感觉
- 玩家能通过行为影响结果
- 游戏具备基础长期目标，即观察成长
- 游戏整体节奏轻量，不制造强焦虑
- 游戏在整体氛围上能唤起 90 后用户对电子宠物时代的怀旧感

### 6.2 本项目暂不追求的体验

首版不以以下目标为优先：

- 深度策略性
- 大量剧情推进
- 大规模收集和养成图鉴
- 社交竞争
- 复杂商业化系统

## 7. 功能需求分析

### 7.1 功能结构概览

首版功能可分为以下 6 类：

1. 宠物初始化与身份设定功能
2. 宠物状态展示功能
3. 玩家交互功能
4. 时间推进与状态变化功能
5. 成长与异常状态功能
6. 存档与恢复功能

### 7.2 详细功能需求

#### FR-01 宠物初始化

功能说明：
用户首次进入游戏时，应获得一只初始宠物，并进入可交互状态。

需求要点：

- 游戏能够识别是否为首次进入
- 首次进入时创建默认宠物数据
- 若已有存档，则直接恢复已有宠物
- 初始状态应处于可照顾、可互动的正常范围

验收关注点：

- 首次进入能成功生成宠物
- 二次进入不会重复初始化覆盖已有存档

#### FR-01A 宠物命名与性别选择

功能说明：
用户在首次创建宠物时，应能够为宠物起名字，并选择其性别，从而增强宠物的专属感和情感连接。

需求要点：

- 首次进入时提供命名入口
- 支持用户输入宠物名字
- 支持选择宠物性别
- 名字和性别信息应参与存档与后续展示
- 若用户未输入名字，可提供默认名字或随机名字兜底

验收关注点：

- 用户可以顺利完成命名和性别选择
- 宠物名字和性别在后续页面中正确显示
- 页面刷新或重新进入后身份信息不会丢失

#### FR-02 宠物状态展示

功能说明：
主界面应清楚展示宠物的关键状态，让玩家知道当前需要做什么。

需求要点：

- 展示宠物名称或标识
- 展示成长阶段
- 展示至少 3 项核心状态
- 展示是否睡眠、是否生病等特殊状态
- 状态显示应尽量直观，避免纯文本堆叠

验收关注点：

- 玩家进入页面后能快速识别宠物当前状态
- 状态变化后界面能及时刷新

#### FR-03 喂食功能

功能说明：
玩家可以通过喂食改善宠物饥饿状态，并获得正向反馈。

需求要点：

- 可通过按钮触发喂食
- 喂食后饥饿值上升
- 可联动提升心情或健康的轻微反馈
- 数值存在上限，不可无限叠加
- 非可交互状态下应有提示

验收关注点：

- 点击喂食后状态正确变化
- 达到上限时行为合理处理

#### FR-04 玩耍功能

功能说明：
玩家可以通过玩耍提升宠物心情，并形成差异化照顾选择。

需求要点：

- 可通过按钮触发玩耍
- 玩耍会提升心情
- 玩耍可能轻微消耗体力或增加饥饿
- 操作应有明显反馈

验收关注点：

- 玩耍行为有明确收益和轻度代价
- 用户能理解该操作的作用

#### FR-05 清洁功能

功能说明：
玩家可以通过清洁行为改善宠物卫生状态，避免异常状态积累。

需求要点：

- 可通过按钮触发清洁
- 清洁后清洁度提升
- 长时间不清洁会带来负面影响
- 清洁行为应与其他行为形成互补

验收关注点：

- 清洁状态能够独立变化
- 用户可以通过清洁干预负面趋势

#### FR-06 睡觉/休息功能

功能说明：
玩家可以让宠物进入休息状态，以恢复精力或触发时间段变化。

需求要点：

- 可主动触发休息
- 休息中部分操作可能受限
- 休息结束后体力恢复
- 休息状态需要明确展示

验收关注点：

- 睡眠状态切换正确
- 睡眠限制逻辑明确，不让用户困惑

#### FR-07 时间推进与状态衰减

功能说明：
宠物状态应随时间自动变化，形成养成驱动力。

需求要点：

- 游戏运行中状态可定时变化
- 离线期间状态也应按规则结算
- 在线衰减与离线衰减应采用不同频率或不同计算策略
- 饥饿、心情、清洁等状态应按不同节奏变化
- 状态变化应有上下限

验收关注点：

- 在线与离线状态结算逻辑应统一遵循同一套规则方向，但允许不同衰减强度
- 状态变化可被用户感知且不失控

#### FR-08 异常状态

功能说明：
当宠物长期未被照顾时，应进入一定的负面或异常状态，以增加规则真实性。

需求要点：

- 可设计疲劳、生病、情绪低落等异常状态
- 异常状态应有触发条件
- 玩家可以通过互动恢复或缓解
- 首版建议采用低惩罚策略，不过度挫败

验收关注点：

- 异常状态触发合理
- 玩家知道如何处理

#### FR-08A 死亡与重新开始机制

功能说明：
当宠物长期处于极端恶化状态且未被及时照顾时，应触发死亡机制，以保留电子宠物玩法中的责任感与情绪冲击；同时需要提供完整的死亡后续流程，保证用户仍能继续游戏。

需求要点：

- 死亡不能由单次短时疏忽直接触发
- 死亡前应存在明确的危险阶段和提示
- 玩家应拥有一定的补救窗口
- 死亡应有可感知的反馈与回顾信息
- 死亡后应允许用户重新开始新一轮养成

验收关注点：

- 死亡条件清晰且可复现
- 玩家能够理解“为什么会死亡”
- 死亡后流程完整，不会让产品进入不可继续状态

#### FR-09 成长阶段变化

功能说明：
宠物应在一段时间和照顾积累后进入新的成长阶段，形成长期动力。

需求要点：

- 至少具备 3 个成长阶段
- 成长条件与时间和照顾质量相关
- 成长后在外观、状态或提示上有可感知变化
- 成长过程可被记录或提示

验收关注点：

- 成长条件稳定可复现
- 成长结果可见且有意义

#### FR-10 存档与恢复

功能说明：
游戏应自动保存关键状态，并在重新进入时恢复进度。

需求要点：

- 本地自动存档
- 页面刷新后可恢复
- 关闭页面后再次进入可恢复
- 存档包含宠物状态、成长信息、最近更新时间等关键字段

验收关注点：

- 存档读取成功
- 时间结算不因存档错误而失真

#### FR-11 提示与反馈系统

功能说明：
游戏应通过文本、图形或轻提示告诉玩家当前发生了什么。

需求要点：

- 操作成功有即时反馈
- 状态过低时有提醒
- 成长发生时有明显提示
- 异常状态有解释信息

验收关注点：

- 玩家不会因缺少反馈而不知所措
- 重要状态变化能够被感知

## 8. 非功能需求分析

### 8.1 易用性需求

- 首次进入用户应在 1 分钟内理解基本玩法
- 核心操作应集中在主界面完成
- 文案和交互提示应尽量直白
- 在表达上兼顾现代清晰度与怀旧氛围，避免完全做成冷冰冰的功能界面

### 8.2 性能需求

- 首屏加载应足够轻量
- 核心交互响应应及时
- 状态刷新不应出现明显卡顿

### 8.3 可靠性需求

- 本地存档不可轻易丢失
- 状态结算逻辑应稳定
- 页面刷新不应破坏当前进度

### 8.4 可维护性需求

- 核心规则逻辑与 UI 层分离
- 状态结构清晰，便于后续扩展
- 关键模块适合编写测试

### 8.5 兼容性需求

- 首版至少兼容主流桌面浏览器
- 页面布局尽量适配移动端宽度

### 8.6 可测试性需求

- 核心状态计算逻辑应可单独验证
- 主要交互流程应可通过手工或自动测试覆盖

## 9. 业务规则与需求假设

### 9.1 初步业务规则

首版建议先采用以下规则方向：

- 宠物拥有多个可变化状态值
- 不同操作影响不同状态
- 状态值有上下限
- 异常状态由低状态积累触发
- 死亡由持续失管和危险状态升级触发，而不是由单次数值归零直接触发
- 成长受时间与照顾质量共同影响
- 离线时间会参与结算，但需限制极端惩罚

### 9.2 待后续文档确认的规则

- 每项状态的具体数值范围
- 状态下降频率
- 生病与恢复的具体触发阈值
- 成长阶段的时间门槛
- 是否允许宠物真正死亡或永久离开

## 10. MVP 范围与优先级

### 10.1 MVP 功能定义原则

MVP 的目的不是做“内容最多”的版本，而是做“最小但完整”的版本。因此功能选择标准为：

- 缺少该功能，核心玩法闭环无法成立
- 缺少该功能，用户无法感受到照顾和成长
- 缺少该功能，游戏无法形成可持续体验

### 10.2 P0 必须实现

- 宠物初始化
- 宠物命名与性别选择
- 主界面展示
- 核心状态展示
- 喂食
- 玩耍
- 清洁
- 睡觉/休息
- 时间推进与状态衰减
- 成长阶段
- 本地存档与恢复
- 基础提示与反馈

### 10.3 P1 建议实现

- 异常状态提示
- 简单成长动画或提示
- 更细致的状态文案反馈
- 离线结算提示面板

### 10.4 P2 后续版本考虑

- 多宠物
- 自定义命名或外观
- 随机事件系统
- 任务系统
- 装扮系统
- AI 对话陪伴
- 成长分支图鉴
- 联网互动能力

## 11. 验收视角下的关键需求

从验收角度看，首版至少需要证明以下几点：

1. 玩家能顺利完成从查看状态到执行互动再到看到反馈的完整闭环
2. 宠物状态会随时间变化，不是静态展示
3. 不同互动对不同状态有差异化影响
4. 游戏在重新打开后能恢复正确进度
5. 玩家能够在一段时间后感知成长变化

## 12. 需求风险与重点关注项

### 12.1 重点风险

- 状态系统太少，导致玩法过于单调
- 状态系统太多，导致首版复杂度失控
- 时间衰减过快，用户产生焦虑
- 时间衰减过慢，用户感觉无事可做
- 成长反馈过慢，用户难以形成期待
- 提示不清楚，用户不知道当前最佳操作

### 12.2 控制策略

- 首版状态数量控制在可理解范围内
- 玩法先做清晰，再做丰富
- 数值节奏在玩法设计和测试阶段重点校准
- 用提示系统降低用户理解成本

## 14. 结论

本项目首版的核心需求已经明确为：围绕一只宠物构建一个轻量但完整的养成闭环，让用户通过少量核心操作持续影响宠物状态，并在时间推进中获得成长与反馈。同时，项目还要通过氛围、交互与反馈唤起 90 后用户对童年电子宠物的记忆。首版将以“可理解、可互动、可成长、可存档、可怀旧共鸣”为核心目标，优先保证闭环成立与体验稳定，而不是追求功能堆叠。
