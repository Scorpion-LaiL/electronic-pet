## 1. 文档目标

本文档用于明确首版 MVP 的测试范围、测试策略、测试阶段、关键测试用例和版本验收标准，保证项目在进入 coding 和联调阶段后有统一的质量判断依据。

本文档重点回答：

- 测什么
- 为什么测
- 怎么测
- 通过什么标准判定首版可交付

## 2. 测试目标

首版测试需要覆盖以下目标：

- 验证核心玩法闭环是否成立
- 验证在线和离线状态结算是否正确
- 验证成长、危险、危急、死亡流程是否稳定
- 验证存档与恢复是否可靠
- 验证 UI 反馈是否足够清晰
- 验证产品是否达到“轻陪伴 + 怀旧感 + 可持续照顾”的基础体验目标

## 3. 测试范围

## 3.1 本次测试纳入范围

首版测试范围包括：

- 宠物初始化
- 命名与性别选择
- 主界面展示
- 五项状态展示
- 四个核心操作
- 在线衰减
- 离线衰减
- 成长系统
- 危险/危急状态
- 死亡与重新开始
- localStorage 存档与恢复
- 提示、弹层和关键交互反馈
- 基础桌面端与移动端适配

## 3.2 本次测试暂不重点覆盖范围

首版可降低优先级或后置验证的内容：

- 高级视觉动画细节
- 大量主题皮肤或换肤方案
- 多浏览器极端兼容性矩阵
- 大量纪念记录展示能力

## 4. 测试类型

首版建议采用以下 4 类测试：

1. `单元测试`
2. `集成测试`
3. `手工功能测试`
4. `体验验收测试`

## 4.1 单元测试

目的：

- 验证纯规则函数是否正确
- 提前发现数值逻辑和边界错误

建议覆盖模块：

- 新宠物创建
- 数值裁剪
- 在线衰减
- 离线衰减
- 四个核心操作
- 成长判断
- 危险/危急/死亡判断
- 存档数据校验

## 4.2 集成测试

目的：

- 验证模块串联后是否正常工作
- 验证“操作 -> 状态更新 -> UI 刷新 -> 自动存档”的完整链路

建议覆盖流程：

- 首次进入创建宠物
- 日常操作后状态更新
- 刷新页面恢复
- 离线后再次进入
- 死亡后重新开始

## 4.3 手工功能测试

目的：

- 验证界面、按钮、提示、弹层等真实使用效果
- 补足自动化测试难以覆盖的页面感知问题

## 4.4 体验验收测试

目的：

- 检查是否真正达到产品目标
- 检查是否唤起怀旧感而不仅仅是功能可用

## 5. 测试环境

## 5.1 开发测试环境

建议环境：

- 本地开发环境
- `Node.js` 当前稳定版本
- `npm` 或 `pnpm`
- Chrome 最新稳定版

## 5.2 验收测试环境

建议至少覆盖：

- 桌面 Chrome
- 桌面 Safari 或 Edge 中的一种
- 移动端模拟宽度

## 5.3 数据环境

由于首版使用 `localStorage`，测试时需重点关注：

- 空存档情况
- 正常存档情况
- 存档损坏情况
- 死亡后记录存在情况

## 6. 测试策略

## 6.1 总体策略

建议采用：

- `规则先测`
- `流程再测`
- `界面最后验收`

顺序建议：

1. 先验证纯逻辑函数
2. 再验证页面和 store 串联
3. 最后进行手工体验测试

## 6.2 风险优先策略

首版最容易出问题的部分应优先测试：

- 在线/离线双频率衰减
- 危险/危急/死亡状态链
- 存档恢复
- 重新开始新宠物流程
- 名字和性别在全流程中的一致性

## 7. 单元测试用例

## 7.1 宠物初始化

### TC-U01 创建新宠物默认值

目标：

- 验证创建新宠物后默认状态是否合法

前置条件：

- 无

步骤：

1. 调用 `createNewPet`
2. 传入名字、性别和当前时间

预期结果：

- 返回完整 `PetRuntimeState`
- 所有状态值在合法范围内
- 名字和性别正确写入
- `isAlive` 为 `true`

### TC-U02 名字为空时兜底

预期结果：

- 系统生成默认名字或合法兜底名称

### TC-U03 性别写入正确

预期结果：

- 性别字段与输入一致

## 7.2 数值与状态工具

### TC-U04 数值下限裁剪

预期结果：

- 任意状态不会低于 `0`

### TC-U05 数值上限裁剪

预期结果：

- 任意状态不会高于 `100`

## 7.3 在线衰减

### TC-U06 在线 10 分钟饥饿衰减

预期结果：

- 饥饿按在线规则下降
- 其他状态按各自节奏变化

### TC-U07 在线短时停留不应直接进入危险

预期结果：

- 正常初始宠物在短时间游玩中不会快速崩坏

## 7.4 离线衰减

### TC-U08 离线 24 小时结算

预期结果：

- 状态明显变差
- 但不应全部接近 0
- 应保留恢复空间

### TC-U09 离线频率比在线更温和

预期结果：

- 同等时长下，离线衰减不直接等价于在线线性累积

### TC-U10 超过离线上限时保护生效

预期结果：

- 超长离线不会无限线性扣减

## 7.5 核心操作

### TC-U11 喂食逻辑

预期结果：

- 饥饿上升
- 上限受控
- 可附带轻微心情提升

### TC-U12 玩耍逻辑

预期结果：

- 心情提升
- 体力和饥饿按规则变化

### TC-U13 清洁逻辑

预期结果：

- 清洁值恢复
- 可缓解卫生相关风险

### TC-U14 休息逻辑

预期结果：

- 体力恢复
- 睡眠状态切换合理

## 7.6 成长系统

### TC-U15 幼年到成长期触发

预期结果：

- 满足时间与照顾条件时可进入下一阶段

### TC-U16 不满足条件时不成长

预期结果：

- 不会误触发成长

## 7.7 危险、危急与死亡

### TC-U17 危险状态触发

预期结果：

- 多项状态过低时进入 `danger`

### TC-U18 危急状态触发

预期结果：

- 危险持续超时后进入 `critical`

### TC-U19 危急状态恢复

预期结果：

- 在补救窗口内恢复后可退出危急状态

### TC-U20 死亡触发

预期结果：

- 满足死亡条件时进入 `dead`
- 同时生成纪念记录

### TC-U21 离线不直接判死

预期结果：

- 离线恢复不会直接跨过补救阶段进入死亡

## 7.8 存档模块

### TC-U22 存档序列化与反序列化

预期结果：

- 可正确保存和恢复数据结构

### TC-U23 存档缺字段兜底

预期结果：

- 缺失字段时不会导致程序崩溃

### TC-U24 非法 JSON 兜底

预期结果：

- 非法内容返回 `null` 或默认恢复流程

## 9. 手工测试检查表

## 9.1 主界面理解度

检查点：

- 玩家首次进入 1 分钟内能否看懂主界面
- 是否能快速理解四个按钮作用
- 是否能看懂状态高低

## 9.2 反馈清晰度

检查点：

- 每次操作后是否有足够清晰的反馈
- 危险和危急提示是否足够明显
- 成长提示是否具有成就感

## 9.3 怀旧感与陪伴感

检查点：

- 是否会让人联想到电子宠物
- 是否有“它一直在等我回来”的感觉
- 是否既怀旧又不过时

## 9.4 离线回归体验

检查点：

- 离线 24 小时后是否合理
- 回归时是否需要一轮照顾，但不至于过度惩罚
- 离线提示是否清楚但不过度打断

## 9.5 死亡体验

检查点：

- 死亡是否足够有情绪重量
- 是否能理解自己为什么失去了它
- 重新开始流程是否顺畅

## 10. 缺陷分级建议

建议使用以下分级：

- `P0`：阻塞主流程，无法继续游玩
- `P1`：核心功能错误，严重影响体验
- `P2`：次要功能问题或体验瑕疵
- `P3`：轻微文案、样式或边缘问题

示例：

- 无法创建宠物：`P0`
- 离线恢复后状态错乱：`P1`
- 危急提示文案不准确：`P2`
- 按钮间距略有问题：`P3`

## 11. 版本验收标准

首版 MVP 建议满足以下验收标准后，方可进入“可交付”状态：

1. 首次进入、命名、性别选择流程全部可用
2. 四个核心操作行为正确
3. 在线衰减和离线衰减都符合规则
4. 离线 24 小时后仍具备合理恢复空间
5. 成长流程可触发且可感知
6. 危险、危急、死亡、重开全链路可跑通
7. 存档与恢复稳定可用
8. 不存在 P0 缺陷
9. P1 缺陷已清零或有明确接受说明
10. 主界面在桌面和移动端基本可用

## 12. 测试执行建议

建议执行顺序：

1. 单元测试先行
2. 集成测试验证主链路
3. 手工测试补足 UI 和体验问题
4. 修复缺陷后执行回归测试

## 13. 回归测试范围

每次修改以下模块后，建议必须执行回归：

- 时间衰减规则
- 存档读写
- 成长逻辑
- 死亡逻辑
- 重新开始流程
- 主界面状态展示
